# Bugs / issues to revisit

- `packages/types/index.ts` / `backend/src/db/schema_glossary.json`: the domain "Glossary" concept (the Athlete's exercise dictionary — exercises → fatigue profiles) collides with "glossary" as the name of the ubiquitous-language file (`CONTEXT.md`). Consider renaming the domain concept to **Exercise Library** (or **Exercise Dictionary**) to remove the collision. See `CONTEXT.md`.
- `AGENTS.md` "Known intentional mismatches" #1 and `docs/ARCHITECTURE.md` "Current rough edges" #1 describe a frontend-local RPE-table / conservative-percent "trend max" that "can legitimately differ" from backend `current_maxes`. Owner ruling: any frontend/backend max disparity is a bug, fixed by a shared function — there is no legitimate divergence to protect. The entry is stale; the "trend maxima" refer to the trend chart's weekly aggregation (`frontend/src/pages/AnalysisPage.tsx`, `frontend/src/utils/rpe.ts`), not a competing max measure. See `CONTEXT.md` Performance invariant.
- `backend/src/controllers/competitionController.ts` `getCompetitions` / `updateCompetitions` read the competition list from `program.competitions` via `health_get_program` (the `if-health` program document) instead of from the dedicated `if-powerlifting-user-competitions` table. Competitions were migrated to their own store(s) (`if-powerlifting-master-competitions` for immutable event data, `if-powerlifting-user-competitions` for editable user-specific entries) but this controller still treats them as a sub-collection of the program document. Parity consequence: there is no `health_list_competitions` function, so the agent cannot list competitions uniformly — it would have to load the whole program and extract the field, the non-uniform path ADR 0001 was meant to prevent. Fix: route through a `health_list_competitions` function backed by the user-competition table. See `CONTEXT.md` Competition Entry / ADR 0001.


- Percentile rankings: not pulling from S3. `analyze_powerlifting_stats` / `powerlifting_ranking_percentile` / `powerlifting_filter_categories` (the `stats` class tools) are supposed to read the OpenPowerlifting export (CSV/Excel) from the powerlifting S3 bucket before computing percentiles. Symptom: returns empty / no data. Likely culprits: S3 bucket name hardcoded vs `var.powerlifting_s3_bucket`, missing IAM permissions on the Fission function's `pl-aws-credentials` secret, or the file path key in the bucket doesn't match. Owner action: invoke one of the stats tools, look at the Fission function pod logs, see if the S3 GetObject is 4xx/5xx.
- Analysis page: no data comes in. Likely related to (a) `health_get_program` / `health_get_phases` returning empty because the if-health program document isn't populated for the logged-in user, OR (b) `analysis_cache_get_window` / `analysis_cache_get_markdown` returning nothing because the analysis cache is empty (the agent hasn't run an analysis yet, or the cache write is failing). Owner action: `kubectl logs -n if-portals -l app=pl-fn-pod-analysis` to see what's happening on the pod.
- Competitions not pulling: matches the entry above — `competitionController.getCompetitions` reads from `program.competitions` (stale path) instead of the new `if-powerlifting-user-competitions` table. Frontend call goes through the backend controller, which calls the Fission `health_get_program` tool; that tool returns the program document with an empty/old `competitions` field. The fix is in `backend/src/controllers/competitionController.ts` per the existing bug entry above — same root cause.
- DOTS calculator: initial value wrong. Likely a frontend form default in `frontend/src/pages/...` (probably an AnalysisPage or Tools page) initialising with a default that doesn't match a valid body-weight/lift combo, OR the calculation function is dividing by a string instead of a number. Owner action: open the page in the browser devtools, check the `value` prop on the body-weight input and the result field; reproduce with a known input.
- Profile crashes: `display.toFixed is not a function for weight value`. The weight field is stored as a string (likely from a DynamoDB number→string round-trip or a `String(...)` somewhere in the data flow) and the frontend assumes `Number`. Owner action: find the profile component in `frontend/src/pages/ProfilePage.tsx` (or similar), wrap the `display.toFixed` in `Number(display).toFixed(2)`, and trace the type back to where the string is being introduced (probably a JSON.parse of a DynamoDB item without the right reviver).

- Stats S3 download (resolved 2026-07-06): added `s3_read: true` to `lambda/pod_analysis/resources.yaml` (so `POWERLIFTING_S3_BUCKET` is injected by `k8s-fission-powerlifting.tf`). Added `_ensure_dataset_downloaded()` helper to all three stats core.py files (`powerlifting_ranking_percentile`, `powerlifting_filter_categories`, `analyze_powerlifting_stats`) — called at the start of `_parse_csvs()`. It uses boto3 to list/download the `openpowerlifting-*.csv` files from S3 to `SANDBOX_PATH` on first call per pod lifetime. The stats tools now work without the CSVs being pre-baked into the image. After ECR rebuild of `pod_analysis`, the 404/empty results on `/api/stats/ranking_percentile` should resolve to actual percentile numbers.
- DOTS initial value (resolved 2026-07-06): the "initialize from program maxes" block in `frontend/src/components/tools/DotsCalculator.tsx` was using `useMemo` (memoized computed value) to call state setters. Replaced with `useEffect` (correct side-effect hook) so the inputs actually populate from `program.meta` when the page loads.
- Profile display.toFixed crash (resolved 2026-07-06): `frontend/src/pages/ProfilePage.tsx` had two `weightValue` functions calling `display.toFixed(1)` and `dots.toFixed(1)` directly on the backend payload. The backend (Fission) returns values that are sometimes `null` or string-typed (e.g. `summary.squat_kg` is a DynamoDB Decimal round-tripped through Flask's `jsonify`), and `Number.isInteger(null)` is `false`, so it would call `null.toFixed()` or `"42.5".toFixed()` — both throw. Rewrote both `weightValue` functions and the DOTS `toFixed` calls to coerce via `Number(...)` + `Number.isFinite(...)` before formatting. Also widened the public profile weightValue signature to accept `number | string | null | undefined`.
- Analysis page error visibility (resolved 2026-07-06): the error block in `frontend/src/pages/AnalysisPage.tsx` was a small red text. Replaced with a card that shows the error text, a Retry button (bumping `analysisRefreshNonce` to re-run the load effect), and a per-section status dump so the next time the user sees a blank analysis page they can tell me exactly which sections are missing/error without me having to guess. The actual root-cause fix for "no data comes in" still needs to be diagnosed against the live env — the page error display is now a real diagnostic tool.
